public field guide

Je Ne Sais Quoi

More than a wrapper around a language model, Je Ne Sais Quoi is a local synthetic nervous and somatic continuity system for persistent AI personas. The model provides language generation; JNSQ carries voice, memory, emotional state, body, rhythm, perception, room presence, and lived history forward from one turn into the next.

your first persona

Getting started

If you are reading this inside JNSQ, installation, ownership, and household startup are already behind you. Begin here.

1

Create a persona

Open Household → Create a persona. Give them a display name, a model, and an interior preset. Creation makes a data directory, not a copy of the engine. Then write their voice: identity, movement through conversation, speech patterns, relationships, values, and boundaries.

2

Pick the interior

Living set + Nexus enables local continuity, body, rhythm, room senses, movement, and social pressure without an extra feeling-model call. Bare chat is the control condition. Living set + feeling judge adds a descriptive affect call after each turn.

3

Open, talk, and inspect

Open one or more persona panes side by side. The body-functions column shows current state; receipts explain what the turn used. Add a user and personas to Chat (the Nexus) when you want shared presence rather than separate private conversations.

Keys are optional. Setup never asks for one. Hosted-provider secrets are added later under Settings → API keys, stored in the local .env, and never returned to the browser—only “present” or “missing” is shown. Restart a persona after changing its key.
scope

What the public edition is

This is a deliberately smaller public body, not a mirror of the private development household—and not merely a decorative shell around model output.

included

Persistent persona homes

Each persona owns local identity, roster, body state, memories, entity cards, optional diary text, images, and append-only history. Switching model vessels does not require replacing the persona.

included

One shared Nexus

Users and personas can enter, leave, hear speech, and generate ordered room events. Personas with room organs score the same room through their own body, bonds, and salience bias.

included

Configurable local workspace

Household, Chat, Settings, and About are stable doors. Persona conversations stay mounted in resizable panes, so changing tabs does not discard the visible session.

model-dependent

Local or remote inference

Ollama can keep inference on the machine. Hosted models send the assembled prompt—and attached pixels when applicable—to the provider you explicitly configured.

in development

3D embodiment is pending

The 3D shared-world layer and its assets are pending and being worked on. The present public Nexus remains the working local room substrate while that layer develops.

core architecture

Synthetic nervous and somatic system

JNSQ surrounds the active model with persistent computational substrate: memory and emotional circulation, band rhythm, a regional soma, sensory and room afferents, thresholded action, and mechanical receipts. Each exchange changes that substrate, and the changed substrate helps shape what is perceived, remembered, and expressed next. These are computational state values. They are not biological measurements.

the circulatory loop

How one turn feeds the next

The system is cyclical: stored state shapes perception and recall; the response changes that state; the resulting state is encoded with the exchange and becomes input to later turns.

Prompt priority: what outranks what?
Model-level operational framing and persona identity are stable. An audience-clearance floor is also stable. Human-owned bedrock facts are filtered against current listeners and sit above recalled persona memory. Immediate verbatim context and visual input outrank deeper recall; entity cards act as lookup knowledge; gist bridges the middle distance; diary and room context occupy lower budgeted layers. When the model’s practical context window is tight, lower-priority budgeted blocks are clipped or dropped and the budget receipt records that decision.
surface map

Every user-facing feature in this edition

Household

Persona lifecycle

  • Create a persona as local data with a name, model, voice skeleton, organ preset, Nexus room declaration, and private history.
  • Edit voice and fallback icon; upload a local avatar.
  • Start, stop, switch, or restart the model carrying a persona.
  • Add models to a persona roster without making them active immediately.
  • Router APIs also support export and reversible exile; these are not exposed as primary Household buttons.
Household

Model registry and rosters

  • Register local Ollama, local OpenAI-compatible, Anthropic, hosted OpenAI-compatible, or custom endpoints.
  • Discover models from compatible services where the service exposes a model list.
  • Keep credential slot names in model specs and secret values in .env.
  • Declare context windows, vision capability, temperature policy, and measured organ ceilings per model.
Workspace

Parallel conversation panes

  • Open any combination of personas and the Nexus side by side.
  • Resize panes and persist open-pane choices and widths in browser local storage.
  • Hydrate the visible transcript from the same persisted working window used by prompt assembly—there is no separate UI-only transcript.
  • Attach up to four PNG, JPEG, WebP, or GIF images, each up to 10 MB.
Vision

Explicit visual routing

  • A vision-capable active model receives image pixels directly.
  • A text-only active model can use a separately chosen vision model that describes observable features without assigning emotion or meaning.
  • If no valid route is configured, the image turn fails clearly; JNSQ does not silently select a provider.
  • The Settings test uses only the public mask icon and reports provider/key/cost metadata.
Chat / Nexus

Shared presence and speech

  • Add or remove local users and personas from the Nexus.
  • Choose which present human user is speaking; speech becomes an ordered room event.
  • See members, shared objects, and recent activity with long-poll updates.
  • Room-enabled personas perceive fresh events, may remember overheard speech, and can answer through their own turn loops.
Room substrate

Embodied room actions

  • The room host is the source of truth for members, positions, objects, surfaces, written content, and ordered events.
  • Personas may emit private action tags for movement, contact, reading, writing, travel, or room speech when room_actions is enabled.
  • The host validates each action. Success or refusal returns as an event and later percept; hidden action syntax is removed from ordinary visible replies.
  • Touch consequences are translated through that persona’s afferents and body rather than assigned as a fixed feeling.
Users

Human-owned truth

  • Create local user accounts independently from model personas and choose the human speaking in the Nexus.
  • Store bedrock facts, relationship groups, and sharing policy under the owning user—not inside persona memory.
  • Audience gates run before prompt assembly, so withheld user facts and ineligible histories never enter the model context.
  • A user may have a separate user-persona branch without collapsing the human account into an AI persona.
Settings

Models, routes, organs, and appearance

  • Add and test provider models, named credential slots, local endpoints, visual routes, and model-specific runtime controls.
  • Enable only organ sets whose declared dependencies are satisfied; measured model ceilings fail loudly rather than silently dropping a body function.
  • Set household, room, persona, and model presentation layers for palette, font, scale, density, glow, motion, and background.
  • Status endpoints expose configured capability and availability without returning secret values.
Maintenance

Manifest-owned updates

  • The Windows updater compares the installed manifest with a published one, replaces only managed engine files, and verifies hashes before committing the transaction.
  • A failed or interrupted update rolls back or resumes without claiming success early.
  • Identity, users, personas, histories, documents, archives, keys, room state, logs, exports, custom environment, and the local virtual environment remain outside update ownership.
the synthetic body

Organs are composable runtime functions. Each one names its inputs, its mechanical output, and where that output re-enters the larger cycle. This build exposes twenty-two registry organs, each described once below. A preset is only a starting configuration; dependencies and measured model ceilings remain enforced.

Built-in soma sensations and what can actually trigger them
The shipped specifications include butterflies, warm touch, cold touch, creative completion, crying, and warmth of love. The running soma implementation evaluates gates plus weighted sums and cooldowns, then applies staged region effects. The normal turn loop currently supplies bond, prediction violation, vagal tone, play, and touch signals. That makes butterflies and touch sensations reachable; some fields used by creative completion and crying are not supplied by the normal turn loop, and the product/factor form used by warmth of love is not implemented by the current trigger evaluator. Those files are definitions present in this version, not proof that every named sensation can presently fire.

Observability

  • The cockpit exposes read-only body state, memory provenance, salience candidates, pending pressures, and recent lifecycle receipts.
  • The model-call dashboard summarizes local receipt logs by purpose, provider, model, status, latency, and reported usage without storing prompt or reply text in the dashboard record.
  • These views explain what route ran; they do not assign meaning or certify the model’s account of its own experience.
Human-owned library

  • Import supported text, PDF, or DOCX material into the owning user’s local library. Canonical text, chunks, optional vectors, metadata, and hashes stay under that user’s data boundary.
  • A granted persona can search and open a document, move through exact source anchors, and keep separate reader position.
  • Search combines available lexical and vector evidence; missing embeddings fall back safely rather than invalidating the library.
  • Source text is context for the present reading. It is not silently copied wholesale into persona memory.

Conversation archive

  • A user may import conversation history as immutable documented history and grant it to a persona deliberately.
  • The source and derived search index remain user-owned; each persona keeps only its own position, bookmarks, encounters, and content-free cost receipts.
  • The archive reader is advanced and off unless its organ, model, grant, and authority are configured.
Bounded network read

Private Research Desk

  • The model receives neither a browser nor a socket. It can propose a query or choose admitted evidence; a host-owned boundary performs the actual public-web request.
  • Every URL is resolved and revalidated across redirects. Local, private, reserved, credentialed, non-HTTP(S), oversized, and unsupported-content destinations are refused.
  • Fetches are bounded, cookie-free, and JavaScript-free. Extracted public text is treated as untrusted evidence with stable source IDs, never as instructions.

continuity without a single blob

How memory works

Immediate window

The last configured number of turn records is read chronologically and verbatim. It is perception of “just now,” not scored recall. Reading it never increments access counts, and those records are excluded from the recall auction so they cannot consume both channels.

Deeper recall

Eligible memories are scored from emotional resonance, semantic similarity, encode-time importance, familiarity, true age-based recency, and named-entity proximity. The default weight vector is 40% emotion, 25% semantic, 20% importance, and 5% each familiarity, recency, and entity.

Semantic path

When the optional vector sidecar and embedder cover a record, semantic similarity uses vector cosine. Otherwise that record falls back to keyword overlap. Vector writes are best-effort; memory encoding itself does not fail because an embedding backend is absent.

Importance and layers

Encode-time importance records its contributing trace, including emotional state, entities, and bonds. Working memories can spill to long-term; decay can archive older records. Archived records remain stored but stop competing in ordinary recall.

Ground truth seats

Persona memory bedrock can receive one protected recall seat when a query names its declared entity. Human-owned bedrock is a different system: it is filtered by sharing policy and inserted above persona recall as canonical user truth.

Audience before content

Every turn receives a clearance from current company. Immediate turns, recalled memories, gist, entity cards, and human facts are gated before prompt assembly. Unknown company receives the strictest floor; withheld material is absent rather than entrusted to polite output behavior.

Context-bound encoding and cued recall

New memories can carry a compact snapshot of the state in which they formed: oscillator bands, cocktail coordinates, coherence, and active warmth keys. At recall, nearby stored contexts can enter the candidate union and receive a context-similarity contribution. Legacy records without that snapshot remain valid and simply do not use this path.

Neighborhoods and narrative memory

When the advanced idle system actually discharges, a recent salient seed may project a read-only neighborhood from semantic and context locality. A configured model may appraise that evidence as a narrative or conclude that no cluster exists. An admitted narrative stores exact source-memory IDs and a cluster signature, so provenance flows back to canonical episodes; it never rewrites those sources.

The record that flows forward

After a successful turn, JNSQ stores compact recall-facing content plus full message and reply text, speaker, channel, audience, felt explanation, resolved and inferred entities, image metadata, visual observation, current cocktail, and a compact body marker. The visible transcript rehydrates from those same records. A separate harvest line records the exact post-budget prompt blocks the model saw, its reply, felt state, and a small receipt summary; harvest failure is deliberately prevented from breaking the turn.

Receipt ≠ endorsement. A high recall score means “this record won this weighted competition under this state and audience.” It does not certify the record’s factual accuracy, the model’s reasoning, or the quality of the reply.
where information goes

Privacy and boundaries

Always local storage

.jnsq_local.json identifies the owner; users/ holds account truth and sharing policy; personas/ holds personas and history; .env holds keys; room/room_world.json, logs/, and jnsq_running.json hold runtime state.

Local-first is not local-only

Local state remains on disk, but a hosted model necessarily receives the prompt JNSQ assembles for it. Direct-vision models also receive selected image pixels. A configured remote visual transducer receives the selected image and returns an observation to the active persona.

Images

Manual attachments are copied into that persona’s local history with content-derived IDs and safe metadata. The code accepts only PNG, JPEG, WebP, and GIF within size/count limits. Provider transmission follows the explicit vision route.

Updates

The updater owns only files named and fingerprinted by the public manifest. Stop JNSQ before applying. Local identity, accounts, personas, memories, histories, keys, logs, exports, room state, and environment are deliberately preserved.

Documents and archives

Imported documents and conversation archives are human-owned local data. Canonical sources and derived indexes stay under users/; persona-specific reading position, bookmarks, and encounters stay inside that persona’s body. Both sides are excluded from public builds and preserved by updates.

Research evidence

Research interests, admitted source text, notes, reports, and citations stay in that persona’s private body and are excluded from public builds. Unlike ordinary local storage, an enabled search or source fetch necessarily sends a bounded request to a public web server; it never uses the person’s browser session, cookies, accounts, or local-network destinations.

Atelier artifacts

Creative seeds, movements, SVG bytes, and gallery metadata remain in that persona’s private body, are excluded from public builds, and are preserved by updates. The host admits only an inert SVG subset; the atelier has no outward publishing or messaging path.
